# Бизнес-процессы VisualERP

В данном документе описываются основные бизнес-процессы и операционные циклы системы VisualERP, принципы их работы, правила изменения остатков и жизненный цикл документов.

---

## 1. Сквозной операционный цикл

Операционный цикл малого производства состоит из следующих шагов:

```txt
Приемка сырья (Purchase Receipt)
   │
   ▼
Остатки на складе (Warehouse Stock)
   │
   ▼
Перемещение в производство (Transfer to Workshop)
   │
   ▼
Заказ на производство (Production Order) ──► Расход материалов (Consumption)
   │
   ▼
Выпуск продукции (Finished Goods Output)
   │
   ▼
Остатки готовой продукции (Finished Goods Stock)
   │
   ▼
Отгрузка клиенту (Shipment)
```

Вспомогательные процессы:
- **Списание (Write-Off)**: для брака, потерь, порчи или технологических потерь.
- **Инвентаризация (Inventory Audit)**: для сверки учетных остатков с фактическими.
- **Аудит изменений (Audit Log)**: для отслеживания всех критических операций.

## 2. Общие правила работы с документами и остатками

### 2.1 Жизненный цикл документов и соответствие API
Все основные складские и производственные документы (`PurchaseReceipt`, `Transfer`, `Shipment`, `WriteOff`, `InventoryAudit`) проходят через единую схему статусов и обрабатываются соответствующими API-эндпоинтами:

1. **Черновик (Draft)**:
   - **Бизнес-логика**: Документ создается, наполняется строками, редактируется. Не создает проводок (`StockMovement`) и не влияет на остатки.
   - **Эндпоинты API**:
     - Создание: `POST /api/<resources>` (возвращает статус `DRAFT`).
     - Редактирование: `PATCH /api/<resources>/:id` (изменяет только черновики).
     - Удаление через API в Phase 3 contract не проектируется; если позже понадобится удаление черновиков, оно должно быть отдельным явно описанным действием.

2. **Проведен (Posted / Shipped / Approved / Completed)**:
   - **Бизнес-логика**: Документ утвержден и зафиксирован. Поля документа блокируются для изменения (immutable). Система автоматически генерирует проводки `StockMovement` и `StockMovementLine`. Влияет на производные остатки и доступность партий (`StockBatch`). Физическое удаление проведенного документа из базы данных **строго запрещено**.
   - **Эндпоинты API**:
     - Проведение: `POST /api/<resources>/:id/post` (или `/ship` для отгрузок, `/approve` для инвентаризаций, `/complete` для производства). Требует обязательной отправки заголовка `Idempotency-Key`.

3. **Отменен (Cancelled)**:
   - **Бизнес-логика**: Перевод из проведенного состояния в отмененное возможен только для исправления ошибок. В текущей реализации складское влияние нейтрализуется через `StockLedgerService.cancelMovement`: связанное движение переводится в `CANCELLED`, а расчет остатков учитывает только движения со статусом `POSTED`. Перед отменой документов, которые могут уменьшить остаток на месте возврата, система проверяет доступное количество.
   - **Эндпоинты API**:
     - Отмена: `POST /api/<resources>/:id/cancel`. Требует обязательной отправки заголовка `Idempotency-Key`.


### 2.2 Расчет складских остатков
Текущий остаток товара (`Item`) по конкретной партии (`StockBatch`) на определенном складе/цехе (`StockLocation`) рассчитывается динамически как сумма всех проведенных движений:

$$\text{Остаток} = \sum (\text{Количество в } StockMovementLine \text{, где } targetLocationId = location) - \sum (\text{Количество в } StockMovementLine \text{, где } sourceLocationId = location)$$

`StockBatch.quantity` в доменной модели понимается как вычисляемое представление. Источником истины остаются только проведенные `StockMovement` и `StockMovementLine`.

## 3. Детализация процессов

### 3.1 Приемка сырья и материалов (Purchase Receipt)
- **Цель**: Отразить поступление сырья, упаковки или комплектующих от поставщика на склад.
- **Действие при проведении**:
  - Создает `StockMovement` с типом `purchase_receipt`.
  - Для каждой строки создает или находит `StockBatch`.
  - Увеличивает остаток на целевом складе (`targetLocationId`).
  - Присваивает статус партии по правилам организации и отраслевого профиля. Конкретные значения по умолчанию должны настраиваться через `IndustryProfile` и будущие правила качества, а не вшиваться в ядро.
- **Сценарий отмены**:
  - Нейтрализует связанное складское движение. Если поступивший материал уже перемещен или списан и на целевом складе не хватает соответствующей партии, отмена блокируется.

### 3.2 Перемещение в производство (Transfer)
- **Цель**: Передать материалы со склада в цех (или между складами) для разделения зон ответственности.
- **Правило**: `sourceLocationId` и `targetLocationId` должны отличаться.
- **Действие при проведении**:
  - Создает `StockMovement` с типом `transfer`.
  - Уменьшает остаток на исходном складе (`sourceLocationId`).
  - Увеличивает остаток на целевом складе/цехе (`targetLocationId`).
  - Перемещает конкретные партии (`StockBatch`), сохраняя сквозную прослеживаемость.
- **Сценарий отмены**:
  - Нейтрализует связанное складское движение. Если на целевом складе уже не хватает переданной партии для возврата эффекта перемещения, отмена блокируется.

### 3.3 Производство (Production Lifecycle)
Процесс производства управляется документом `ProductionOrder` (Заказ на производство) и имеет расширенный жизненный цикл:

```txt
 запланирован (Planned)
        │
        ▼
   в работе (In Progress) ◄─── Передача материалов в цех
        │
        ├──► Регистрация расхода (Production Consumption)
        ├──► Регистрация выпуска (Finished Goods Output)
        ▼
   завершен (Completed) / отменен (Cancelled)
```

1. **Запланирован (Planned)**:
   - Рассчитываются плановые потребности в сырье (`ProductionOrderLine`) на основе выбранной версии `BOM` (Рецептуры).
   - Если `BOM` не указан при создании заказа, сервер может привязать единственную активную `BOM` для той же организации и выпускаемого изделия. Если активных `BOM` несколько, заказ не создается до устранения конфликта или явного выбора `bomId`.
   - Если подходящей активной `BOM` нет, заказ остается допустимым без `BOM`; при завершении потребуется явный фактический расход сырья.
   - Резервирует материалы на складе (опционально, виртуальный резерв). Физические остатки не меняются.
2. **В работе (In Progress)**:
   - Цех начинает работу. Сырье перемещается со склада в цех с помощью документа `Transfer`.
3. **Расход сырья (Production Consumption)**:
   - Фиксируется фактическое использование сырья в цеху (документ `ProductionConsumption`).
   - Фактический расход может отличаться от планового по BOM (перерасход или экономия).
   - Создает `StockMovement` с типом `production_consumption`. Списывает сырье из цехового расположения (`sourceLocationId = workshop`).
4. **Выпуск продукции (Finished Goods Output)**:
   - Фиксирует появление готовой продукции или полуфабриката (документ `FinishedGoodsOutput`).
   - Создает `StockMovement` с типом `production_output`.
   - Создает новую партию готовой продукции (`StockBatch`). Начисляет остаток на целевой склад готовой продукции.
5. **Завершен (Completed)**:
   - Заказ закрывается. Рассчитывается фактическая себестоимость на основе списанного сырья и выполненного выпуска. Корректировка планового сырья больше недоступна.

---

### 3.4 Отгрузка продукции (Shipment)
- **Цель**: Отразить продажу и физический вывоз готовой продукции клиенту.
- **Статусы документа**: `draft` -> `shipped` -> `cancelled`.
- **Действие при проведении**:
  - Создает `StockMovement` с типом `shipment`.
  - Списывает продукцию с указанного склада (`sourceLocationId = warehouse`).
  - Уменьшает количество в выбранной партии (`StockBatch`).
- **Сценарий отмены**:
  - Возвращает продукцию на склад.

---

### 3.5 Списания (Write-Off)
- **Цель**: Отразить брак, порчу сырья, повреждение упаковки или выдачу образцов.
- **Правило**: Списание всегда уменьшает остаток и требует обязательного указания причины (`reason`) и ответственного сотрудника.
- **Причины списания**:
  - `technological_loss` — технологические потери (нормативные потери при запуске оборудования, очистке емкостей).
  - `defect` — брак производства.
  - `damage` — порча/повреждение при хранении.
  - `inventory_correction` — списание по итогам инвентаризации (недостача).
  - `sample` — отбор проб для лаборатории или демонстрации.
  - `other` — иное (с обязательным комментарием).
- **Действие при проведении**:
  - Создает `StockMovement` с типом `write_off`.
  - Списывает указанные позиции и партии с выбранного склада/цеха.

---

### 3.6 Инвентаризация (Inventory Audit)
Инвентаризация предотвращает бесконтрольное ручное изменение остатков:

1. **Создание документа**: Выбирается склад/цех (`locationId`) и фиксируется время аудита.
2. **Расчет учетного остатка**: Для переданных строк система рассчитывает ожидаемое количество (`expectedQuantity`) по проведенным движениям склада на текущий момент.
3. **Фактический подсчет**: Аудитор передает обязательный список фактических строк в действие `count`.
4. **Расчет расхождений**: Система считает расхождение: `discrepancy = actual - expected`.
5. **Утверждение**:
   - Если расхождений нет — документ просто закрывается.
   - Если есть недостача (разница < 0) — автоматически создается `StockMovement` (тип `inventory_adjustment`), который списывает недостающее сырье/продукцию.
   - Если есть излишек (разница > 0) — автоматически создается `StockMovement` (тип `inventory_adjustment`), который приходует излишки на выбранное место хранения и указанную партию, если она передана.
6. **Отмена**: Сторнирует корректирующее движение инвентаризации. Для уже утвержденных излишков перед отменой проверяется, что нейтрализация не создаст отрицательный остаток.

## 4. Панель управления (Dashboard)

Панель управления (`GET /api/dashboard`) агрегирует ключевые операционные показатели организации в реальном времени:
1. **Сводка по складу (Stock Summary)**: Отображает количество активных позиций, складов и партий с положительным балансом, а также суммарные остатки товаров в разрезе типов.
2. **Товары с низким запасом (Low Stock)**: Отображает список товаров, остатки которых упали ниже критической отметки (на текущем этапе возвращает пустой список, так как минимальные пороги запасов не хранятся в схеме БД).
3. **Сводка по производству (Production Summary)**: Показывает количество заказов на производство по статусам, объем выполненного производства за текущий месяц и список последних закрытых заказов.
4. **Сводка по отгрузкам (Shipment Summary)**: Показывает объем отгрузок клиентам по статусам, объем отгрузок за текущий месяц и последние выполненные поставки.
5. **Сводка по списаниям (Write-off Summary)**: Отражает статистику списаний по статусам, объемам текущего месяца и причинам списаний.
6. **Документы в ожидании (Pending Documents)**: Отображает счетчики документов, требующих внимания (черновики документов поступления, перемещения, списания, отгрузки, плановые заказы на производство и инвентаризации в статусе `COUNTED`).
7. **История аудита (Recent Audit Events)**: Показывает 10 последних записей из системного журнала аудита изменений для контроля действий пользователей.
