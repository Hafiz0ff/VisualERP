# Модули системы VisualERP

Инфраструктура VisualERP разработана по модульному принципу. Это позволяет системе оставаться легковесной для небольших цехов и постепенно расширяться по мере роста бизнеса без необходимости разделения (форка) кодовой базы.

---

## 1. Архитектурная формула

$$\text{VisualERP Core} + \text{Industry Profiles} + \text{Optional Modules}$$

- **VisualERP Core (Ядро)**: содержит базовые сущности (Организация, Пользователи, Роли, Права доступа, Единицы измерения, Товары), явные операционные сущности `Warehouse` и `Workshop`, а также общую шину движений запасов (`StockMovement`) через `StockLocation`.
- **Industry Profiles (Отраслевые профили)**: наборы преднастроенных конфигураций терминологии, категорий и рекомендуемых модулей для конкретных отраслей (например, `dry_mixes`, `food`, `dairy`, `tools`).
- **Optional Modules (Опциональные модули)**: функциональные расширения, которые могут включаться или выключаться для каждого арендатора (организации).

---

## 2. Спецификация модулей

### 2.1 Ядерные (Базовые) Модули
Эти модули всегда активны и не могут быть отключены:
- **Core Settings & Tenancy**: Управление организациями, их базовыми валютами, часовыми поясами.
- **Identity & RBAC (Users/Roles)**: Аутентификация пользователей, распределение прав и ролей в рамках организации.
- **Items & Units (Master Data)**: Управление номенклатурой товаров (`Item`), категориями (`ItemCategory`), единицами измерения (`Unit`) и правилами их конвертации.
- **Audit Log**: Сквозное логирование всех чувствительных действий (изменение прав, проведение документов).

### 2.2 Опциональные Модули (MVP)
Эти модули подключаются через настройки организации (`ModuleConfig`):

#### 1. Warehouse (Управление складом)
- **Функционал**: Учет остатков на складах, создание поступлений от поставщиков (`PurchaseReceipt`), перемещение товаров между складами (`Transfer`), проведение инвентаризаций (`InventoryAudit`).
- **Зависимости**: Зависит от *Master Data*.

#### 2. BOM / Recipe (Спецификации и Рецептуры)
- **Функционал**: Создание и версионирование спецификаций (`BOM`) и их строк (`BOMLine`) для готовой продукции и полуфабрикатов.
- **Зависимости**: Зависит от *Master Data*.

#### 3. Production (Производство)
- **Функционал**: Создание производственных заказов (`ProductionOrder`), учет фактического расхода сырья (`ProductionConsumption`) и регистрации выпуска (`FinishedGoodsOutput`).
- **Зависимости**: Зависит от *Warehouse*, *BOM / Recipe* (опционально, см. правила отключения).

#### 4. Shipments (Отгрузки)
- **Функционал**: Документы отгрузки готовой продукции покупателям (`Shipment`) и справочник клиентов (`Customer`).
- **Зависимости**: Зависит от *Warehouse*.

#### 5. Write-offs (Списания)
- **Функционал**: Оформление брака, порчи, отбора проб и других непроизводственных потерь (`WriteOff`).
- **Зависимости**: Зависит от *Warehouse*.

#### 6. Reports (Отчетность)
- **Функционал**: Визуальные отчеты по остаткам, оборачиваемости, эффективности производства и отгрузкам.
- **Зависимости**: Собирает данные из всех активных транзакционных модулей.

## 3. Взаимосвязь и правила отключения модулей

Включение и выключение модулей производится через запись в таблице `ModuleConfig` для конкретной организации.

### 3.1 Принцип "Мягкой деградации" (Graceful Degradation)
Выключение модуля не должно приводить к ошибкам компиляции или поломке базы данных. Вместо этого система адаптирует свое поведение:

1. **Если отключен модуль BOM / Recipe**:
   - Модуль **Production** продолжает работать, но заказы на производство (`ProductionOrder`) создаются без указания BOM.
   - Расход материалов при производстве вносится оператором вручную по факту, так как нет нормативных данных для автоматического заполнения.
   - Пользовательский интерфейс скрывает вкладку "Спецификации".

2. **Если отключен модуль Shipments**:
   - Готовая продукция накапливается на складах. Ее уменьшение производится только через ручные списания (`WriteOff`) или инвентаризацию (`InventoryAudit`).
   - Кнопка "Создать отгрузку" и реестр отгрузок скрыты в UI и заблокированы в API.

3. **Если отключен модуль Production**:
   - Система работает как простой складской учет (приход -> хранение -> перемещение -> списание/отгрузка).
   - Все сущности, связанные с производством, скрыты.

### 3.2 Контроль на уровне API и UI
- **API Guard**: При попытке сделать запрос к эндпоинту отключенного модуля (например, `POST /api/boms` при выключенном модуле BOM), сервер возвращает ошибку `403 Forbidden` с кодом `MODULE_DISABLED`.
- **UI Dynamic Layout**: На этапе загрузки приложения клиент получает список активных модулей (`ModuleConfig`) и скрывает соответствующие разделы меню, кнопки действий и виджеты отчетов.

## 4. Отраслевые конфигурации по умолчанию

При создании новой организации и выборе Отраслевого профиля (`IndustryProfile`), система автоматически включает рекомендованные модули:

| Отраслевой профиль | Warehouse | BOM / Recipe | Production | Shipments | Write-offs | Примечание |
| :--- | :---: | :---: | :---: | :---: | :---: | :--- |
| **Сухие смеси (dry_mixes)** | Да | Да | Да | Да | Да | Классический полный цикл |
| **Пищевое (food)** | Да | Да | Да | Да | Да | Требует контроля партий и сроков годности |
| **Молочное (dairy)** | Да | Да | Да | Да | Да | Усиленные требования к срокам годности и прослеживаемости |
| **Мясное производство (meat_processing)** | Да | Да | Да | Да | Да | Важны партии, охлаждение и контроль потерь |
| **Мебель (furniture)** | Да | Да (Состав) | Да | Да | Да | BOM используется как подетальный состав изделия |
| **Инструменты (tools)** | Да | Да | Да | Да | Да | Учет комплектующих и сборочных этапов |
| **Текстиль (textile)** | Да | Да | Да | Да | Да | Важны полуфабрикаты, партии и межцеховые перемещения |
